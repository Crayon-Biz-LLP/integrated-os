# Fallback Contracts

Single authoritative source for degraded-mode behavior when LLM calls or critical
dependencies fail. The test suite (`tests/sim/`) asserts against these contracts.

---

## C3 — Classification LLM Failure

**Governing code**: `core/webhook/classify.py:classify_intent()`

**Triggers**:
- `generate_content_with_fallback()` raises (network error, timeout, provider 5xx)
- Rate limiter reports wait > 3s (D5)

**Output**: `SAFE_HOLD_CLASSIFICATION` constant from `core/llm/constants.py`:

```python
{
    "intent": "NOTE",
    "confidence": 1.0,
    "entity": "INBOX",
    "title": "Fallback Note",
    "time_context": "",
    "clarification_question": "",
    "receipt": "Message vaulted safely (AI classification temporarily unavailable).",
    "possible_intents": [],
    "reasoning": "safe_hold"
}
```

**User-facing behavior**:
- Message is vaulted as a NOTE — embedded into memories, never enters the
  task/completion/project-update pipeline
- No Telegram error or alert shown to user
- `audit_log_sync("webhook", "ERROR", ...)` is written with the exception detail
- The receipt is the only user-facing signal: "Message vaulted safely..."

**Test assertions**:
- `classify_intent()` returns `SAFE_HOLD_CLASSIFICATION` dict exactly
- Caller routes as NOTE (no tasks created, no completions processed)
- Message is persisted in `memories` with `intent: NOTE`

## K2 — Thread Routing Failure

**Governing code**: `core/lib/conversation.py:resolve_thread()`

### Inner catch (entity resolution failure)

**Triggers**: `_fetch_entity_candidates()` raises (deterministic resolver or LLM
fallback both fail)

**Output**: Logs WARNING, falls through to priority 4 (general fallback —
reuses existing general thread or creates new one)

**User-facing behavior**: None. The user sees no difference from a general-thread
reply. No receipt or error is generated by the thread router itself.

### Outer catch (full routing failure)

**Triggers**: Any exception in `resolve_thread()` not caught by inner handlers
(e.g., DB connection failure, unexpected error in thread management)

**Output**:
- Logs `audit_log_sync("routing", "ERROR", "Thread routing failed, falling back to new session: ...")`
- Returns `(str(uuid.uuid4()), None)` — brand-new UUID session, no anchor,
  no conversation_history row
- `get_or_create_session()` receives this and calls `get_history(new_uuid)`
  which returns `[]` (empty history pairs)
- The caller in `handler.py` proceeds with the new UUID and empty history
  as if it were a normal first-time conversation

**User-facing behavior**:
- No user-visible artifact from the fallback itself
- No "new session started" receipt
- No error message sent to user
- If classification also fails, C3's SAFE_HOLD receipt is emitted
  (the two contracts compose independently)
- If classification succeeds, the normal receipt is sent

**Test assertions**:
- Return value is `(uuid4_string, None)` where uuid4_string has UUID format
- No `conversations` row is created for the fallback session UUID
- No Telegram message is sent by the fallback path
- `audit_logs` contains an ERROR-level entry with `"Thread routing failed"` message
- The caller handler continues normally with `history == []`

## M5 — Expired Memory Sweep Failure

**Governing code**: `core/pulse/sentinel.py` — M5 piggyback

**Retry policy**: Immediate retry once per item (no backoff). If both
attempts fail:
- Per-item WARNING is emitted via `audit_log_sync("sentinel", "WARNING", ...)`
- Sweep continues to next item (fail-open, graceful degradation)
- If >50% of items in the batch failed, a summary WARNING is emitted
  at the end of the batch

**Test assertions**:
- First run deletes expired memories and cleans up retrieval index
- Per-item failure produces exactly 1 WARNING audit log per failed item
  (not per attempt — both attempts collapse to one warning)
- Batch failure >50% produces an additional summary WARNING
- Second run (idempotent) produces NO new state mutations:
  no new `memories.delete()`, no new retrieval index cleanup calls
  (harmless audit_log read queries are expected)
